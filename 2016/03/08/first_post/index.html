简单工厂模式

发表于 2016-03-08   |   分类于 设计模式   |     |   阅读次数

工厂模式

简单工厂模式

举例：有三个类：苹果类Apple、香蕉类Banana、橘子类Orange，他们都继承自抽象水果类Fruit
还有一个工厂类FruitFactory，里面有个newInstance()方法，根据newInstance()的参数返回不同的水果对象；

工厂模式

简单工厂模式的弊端：工厂类负责每个具体实例的创建，当有新的实例增加时，需要修改工厂类的代码。
举例：抽象出一个工厂接口FruitFactory，接口里面抽象方法newInstance()返回产品对象，有三个具体的工厂类AppleFactory、BananaFactory、OrangeFactory都继承FruitFactory。其中每个具体的工厂类分别生产具体的水果类：苹果类Apple、香蕉类Banana、橘子类Orange。

抽象工厂模式

虽然工厂模式解决了简单工厂模式中工厂类职责太重的问题，但是由于每个工厂类只创建一类产品系统会产生大量的工厂类。
因此可以将一组相关产品组成一个‘产品族’，由一个工厂来统一生产。
举例：有一个工厂接口Factory，有两个抽象方法createPhone()返回Phone对象还有一个方法createComputer()返回Computer对象；有两个具体工厂如三星工厂SumFactory、联想工厂Factory；三星工厂生产三星手机和三星电脑，联想工厂生产联想手机和联想电脑；

单例模式：

整个应用中只有一个实例存在，从而避免创建果果实例而浪费空间；

  1. 饿汉式：对象随着类的创建而创建；
  2. 懒汉式：调用getInstance()方法时才创建对象，并且使用Double check 模式创建，两次非空判断并且加上同步操作

    多例模式：

    一个类可以创建多个相同实例，并且都是该类本身。

    建造者模式：

    适用于当我们要创建的对象比较复杂的时候，将构造和表示相分离，从而使一个构造过程可以创建不同表示的产品。
    适用场景：
  3. 对象的属性需要按照一定的顺序赋值才有意义，每次调用设置属性的方法都可以返回该对象本身从而实现链式调用；
  4. 对象的创建过程中会使用到系统中的其它对象，而这些对象在产品对象中不易得到；
    建造模式组成：
  5. 导演者Director
  6. 抽象构造者Builder
  7. 具体构造者ConcreteBuilder
  8. 产品Product
